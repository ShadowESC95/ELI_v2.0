# Test Coverage — Methodology, Measurement, and Exclusions

This document describes how ELI's automated test coverage is measured, how to
reproduce the figure, and which code is excluded from the line-coverage metric
**with a stated, auditable rationale**. It is written for external review
(security, privacy/GDPR, and procurement audits).

Coverage is **measured, reproducible, and honest** — no line is counted that a
test does not genuinely execute, and no exclusion is undocumented.

---

## 1. How to reproduce the number

```bash
bash scripts/coverage_full.sh
```

Outputs `artifacts/coverage_report.txt` (text, per-file) and
`artifacts/coverage_html/index.html` (browsable, line-by-line). Configuration is
in `.coveragerc`. The figure below is regenerated by this command on any checkout.

## 2. Why coverage is measured in three lanes

The fast unit suite deliberately **mocks the heavy runtime dependencies**
(`pydantic`, `llama_cpp`, `torch`, `faiss`, `PySide6`) in `tests/conftest.py` so
that thousands of cognition/logic tests run in seconds without a GPU, a model, or
a display. A consequence is that, under that suite, the real web server cannot be
imported, a real model cannot be loaded, and the GUI cannot be constructed — so
their genuine execution must be measured on **separate, un-mocked lanes** and
combined (`coverage combine`):

| Lane | Deps | Exercises | Invocation |
|------|------|-----------|------------|
| 1. Unit suite | mocked | cognition/logic breadth + core paths | `pytest tests/` |
| 2. Web server | **real FastAPI** | `api/server.py` auth/RBAC + endpoints | `pytest tests/test_api_server.py --noconftest` |
| 3. Live engine | **real GGUF model** | full pipeline: router → agent bus → grounding → GGUF → governance | `pytest tests/test_engine_integration_live.py --noconftest` |

Lanes 2 and 3 self-skip under the mocked suite, so they never destabilise the
fast run; they are exercised for real on their own lanes and folded into the
combined report.

## 3. Coverage discipline (no gamed tests)

Assertions target **behaviour the code guarantees**, not incidental line
execution. In particular, the live-model lane asserts *structure* — routed
action, non-empty governed reply, result-shape contract, session continuity,
safe handling of empty/garbage input — and never an exact model string (which
would couple the test to a specific model's output rather than to the code).

## 4. Security- and privacy-critical coverage

The paths most relevant to a security/GDPR review are covered directly and
deeply:

- **Authentication / RBAC** (`api/server.py`): bearer parsing, role hierarchy,
  loopback detection, every `_resolve_principal` branch (tokenless owner /
  correct token / wrong token → fail-closed / no-config → deny), and the live
  401/403/200 gate through the real dependency graph.
- **Offline-by-default enforcement** (`eli/core/netguard`): the socket failsafe
  and egress recording are unit-tested.
- **Data isolation**: the test harness asserts every DB path resolves inside a
  throwaway directory (`tests/conftest.py::_assert_db_isolated`) — a test can
  never touch real user data.

## 5. Documented exclusions (justified, not hidden)

Some code cannot be executed in an automated, headless CI context. It is
**excluded from the line metric with rationale**, not faked:

| Area | Statements | Why excluded from automated line coverage |
|------|-----------:|-------------------------------------------|
| Interactive desktop GUI (`eli/gui/EliMainWindow` full construction) | ~12k | Blocks on interactive display/device/audio init; full-window construction does not terminate headless. Testable GUI *units* (pure helpers, small widgets) are still covered; the interactive window is validated manually and via the run-book. |
| GPU/model-load internals beyond the live smoke lane | — | Depend on specific hardware/VRAM; the model-agnostic load path is exercised with a small model on the live lane, but exhaustive per-backend paths (CUDA/Metal/CPU-offload variants) require the matching hardware. |
| Hardware/OS device I/O (real microphone, camera, speakers, window manager) | — | Require physical devices / a live desktop session; validated on-device, not in headless CI. |

These exclusions are standard and defensible: automated line coverage measures
**headless-executable logic**; interactive-UI and hardware I/O are validated by
other means (manual test run-book, on-device acceptance).

## 6. Current measured figure

Regenerate with `scripts/coverage_full.sh`; the latest committed run is recorded
in `artifacts/coverage_report.txt`. Headline and per-subsystem numbers are in
that artifact so the figure is always the *measured* one, never a stale claim in
prose.
